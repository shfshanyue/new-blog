很多人一开始觉得 fetch 够用了，真上生产才发现麻烦一堆：请求挂着不返回、重试把接口打爆、错误里只有状态码没有正文。

问题往往不是「会不会 await」，而是缺一层薄薄的封装，把超时、重试、错误解析写进同一条流水线里。

1 总超时要跨重试算。只给单次请求设 deadline，重试三次就可能拖到用户以为页面死了。把「整次调用最多等多久」单独设出来，和「单次重试几次」放在一起想，才不容易误伤。

2 钩子签名变了就老老实实对一遍。升级大版本时别只改 import，发布说明里关于前缀、基址合并、查询参数和旧行为差异的部分，最好对着现有调用扫一遍。省下来的半小时会在联调时加倍还回来。

3 空响应和 204 别假装是 JSON。有些接口成功时 body 就是空的，硬 parse 只会得到莫名其妙的异常，排查时还容易误判成网络问题。在封装层把「有没有 body」和「要不要当 JSON」说清楚，比在每个调用点写 if 省心。

4 边缘和浏览器共用一套客户端时，注意运行时的版本门槛。新特性往往跟着 Node 或运行环境走，本地和部署不一致时，最先坏的是 CI 或预览环境。

封装不是为了炫技，是为了把「会咬人的默认值」关进一个地方。你改一处，全项目的请求行为一起升级，这才是省心的做法。
