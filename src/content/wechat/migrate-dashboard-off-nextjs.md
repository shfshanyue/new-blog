我前阵子刷到一篇工程复盘，讲他们怎么把线上控制台从 Next.js 整坨迁出去。听着像大手术，但拆开看，节奏其实很克制。

动机也实在，构建有时候能干到十分钟，流水线门口天天排队。产品又是面板形态，状态主要在浏览器里跑，WebSocket 也不少，不太需要那种「页面先到服务器转一圈」的范式。

他们没搞大爆炸，两条 PR 做完。先把 next/image、老路由 API 这些绑定一点点拔掉，让业务别黏死在框架上。第二条 PR 才在两百多条路由上一口气换到 Vite 加 TanStack Router，再用 Nitro 把重定向和安全头收一收，发布链路还是可控的。

做完的好处很朴素，本地和 CI 变快，静态资源按哈希拆，回源也更局部。代价也写得直白，图片优化和 SEO 小工具要自己补，新栈成熟度只能团队自己扛。

你要是控制台类项目，别把用不用大框架当成信仰战，把构建时间、发布风险和你要的交互形态摊开算一遍，答案会清楚很多。

1 先拆框架绑定，再做大路由迁移，比一夜重写稳。

2 用 Nitro 这类层统一重定向和安全头，线上变更更可控。

3 提前列清楚会失去哪些内置能力，别上线才发现坑。
