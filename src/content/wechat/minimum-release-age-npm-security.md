最近几次 npm 上的幺蛾子，套路都差不多：名字眼熟、版本号新，你一装就中招。

问题不全在「你会不会点安装」，还在于恶意发布往往抢在刚上线的那几个小时。安全社区和扫描工具也需要一点时间才能把异常标出来。

最小发布年龄这类策略，说白了就是：新版本在仓库里躺满一定时间之前，你的 CI 或本地默认不去拉它。pnpm、Yarn、Bun 等都能配，思路相近，只是开关名字和写法不一样。

它当然拦不住所有风险。作者自己作死、内部被钓鱼，照样会出事。但在「刚发布就被投毒」这条线上，多等一等往往就多一层缓冲。和锁文件、来源校验、漏洞扫描放在一起，比单押其中一条要稳得多。

麻烦在于另一端：真有紧急安全补丁，你也不能被自己的规则卡死。团队里最好约定热修通道，哪些包可以缩短等待、谁有权批、怎么留审计记录。安全和平时的发版节奏要对齐，而不是两套各说各话。

如果你负责平台工程或基建，可以把这条策略写进文档里，和新同事 onboarding 时讲清楚。装包不是点点鼠标那么简单，背后是一整条链路的习惯。
